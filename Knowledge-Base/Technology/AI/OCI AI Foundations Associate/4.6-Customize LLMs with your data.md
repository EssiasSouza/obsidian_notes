# 4.6-Customize LLMs with your data
In this lesson, let us look at how you can customize LLMs to work with your own data. So the framework we are going to discuss here is as follows.

So we have two axes, the horizontal and the vertical axis. On the horizontal axis, you have context optimization. Here, we are providing more specific and detailed information for the LLM to draw from. So example could be user-specific information, like orders users have made on a website. On the vertical axis is LLM optimization. Here, we are adapting the LLM to accomplish specific tasks or work within a given domain. So this could be your fine-tuning the model on legal domain.

So the first and the most obvious place to start with is prompt engineering. It's the fastest, the quickest. You can learn quickly, test, iterate quickly. Then if you need more context, you could use something like Retrieval Augmented Generation. If you need more instruction-following, then you can use fine-tuning. And these methods are all additive. So you might end up using all of them.

Let us look at each of them in a little bit more detail. So starting with Retrieval Augmented Generation, here what we do is the language model can query enterprise knowledge bases. So these could be vector database. It could be wikis, et cetera to provide grounded response. And what we mean by grounded is the generated text is grounded in a document if the document supports the text.

So on the left-hand side, you can see a person chatting with a chatbot. And the person wants to return the dress they just bought. And the chatbot says, yes, sure. You can return. But I need to check the return policy. And then the chatbot goes to an enterprise database, looks up the chatbot return policy, which says 30 to 90 days, and items cannot be on sale. And then converses with the user, saying these are the rules. And the user says, I don't know if my item was on sale.

So they upload a receipt. It's using some kind of a vision service, extracting the price and the date when the item was sold, going, again, back to an enterprise database, returning the data and saying that your item can be returned. Now, this is an example of a RAG implementation. And you can see how the answers are grounded. It's going back-- the chatbot is going back to a vector database and figuring out the return policy, et cetera, the rules for the return policy.

Now, RAG does not require fine-tuning. So that's a huge advantage. Literally what you are doing with RAG is there are two major components. One is retrieval, where you are searching over a corpus of information relevant to respond to a user, like you see here. And the second is augmented generation. You are using that information retrieved to form a more informed response.

Now, using RAG, you can give LLM access to private knowledge that it otherwise would not have had, so the return policy in this case. By letting the model search over private database and use that information to form responses, the accuracy and usefulness of the model changes drastically, so as you can see on this example here. So that's an example of RAG implementation.

Now, when we talk-- when we talk about LLM fine-tuning and inference, very simply, fine-tuning, the way it works is we take a pre-trained foundational model and provide some custom data and fine-tune that model, train that model, on that custom data. And we end up with this custom data.

And then for inference, basically we give new text as an input. And the model generates output based on the knowledge it has learned during pre-training and fine-tuning. So based on both of that, the knowledge it has gained, it responds. So that's the inference for the fine-tuned model.

How does really fine-tuning work? Well, what you are doing with fine-tuning is you are optimizing the model on a smaller domain-specific data set, as you can see here. You would use fine-tuning when the pre-trained model doesn't perform your tasks well or when you want to teach it something new. So it could be a unique domain. It could be new skills, tasks, unique style.

So you take the pre-trained model, you infuse this knowledge, and then you end up with a custom model or a fine-tuned model. A fine-tuned model is basically adapting to a specific style and tone and learning domain-specific words and phrases, as is shown here.

Now, we are not going to get into advanced concepts like how the weights are updated, et cetera. But in OCI, there is a method called T-Few fine-tuning, which inserts new layers and selectively updates only a fraction of the model's weight, as you can see here. Now, doing this, it significantly reduces the training time and the cost compared to updating all the layers of the pre-trained model. And this is more of an advanced concept which we discuss in other courses.

There are two main benefits of fine-tuning. The first is you are improving the model performance on specific tasks. It can better-- because you are customizing the model to domain-specific data, it can better understand and generate contextually relevant responses. So that's number one.

Number two is you are also improving the model efficiency. You are reducing the number of tokens. And you're condensing the expertise of a large model into a smaller, more efficient model. So those are two advantages, both improving model performance and also model efficiency.

So the next question which comes up, an obvious question, is should you use one or another and when to use which method. Well, this is, again, a framework which you can use. Prompt engineering is-- the main advantage, it's very simple to use. There is no training cost. And when should you use it? When your LLM already understands topics that are necessary for the text generation.

RAG system, on the other hand, is really useful to use when the data changes rapidly or when you want to mitigate hallucinations by grounding answers, as you saw in the chatbot example. And the advantages are quite obvious. You have you're grounding the results. You are accessing the latest data. The disadvantage is you require a compatible data source. If your data source is not good, your retrieval quality is not going to be good. And it's more complex to set up.

And finally, fine-tuning is needed when your LLM does not perform well on a particular task and the data required might be too large. So you would use fine-tuning. The advantage, as we discussed on the previous slide, model performance improves. Model efficiency improves. But the disadvantage is it requires a labeled data set, which can be expensive to acquire and time-consuming. And it requires more resources to do.

All right. So finally, let me leave you with this kind of framework, which we talked at the beginning of this lesson, and discuss how you might end up using all of them. So a typical journey would look something like this.

You start off at the bottom corner. You have a prompt and an evaluation framework. And you figure out what the baseline is. Then you give few short examples of input/output pairs you want the model to follow. And if you're happy here, probably just might do prompt engineering and not do anything other than that.

Now, let us say these few short examples increase your model performance. And now you can hook the model to an enterprise knowledge base and create a RAG system, as we discussed. So you might actually do it. And your results look good. Your evaluator is the baseline. And probably might just stop here.

But if not, let's say the model output is not coming out in the format or the style that we want, then you can fine-tune this RAG-optimized model. So you could optimize it and then fine-tune this model. And maybe you are happy there.

But let's say if the retrieval with this model is not good. So you can optimize the retrieval. And you can optimize the RAG system, and so on and so forth. You can see how this works iteratively. And you end up using all these three mechanisms. So this is a good framework to think about when you are thinking about prompt engineering retrieval, augmented generation, and fine-tuning.

So that's pretty much it. It's a very interesting area. It's something which is very useful, this kind of framework, to think about when you are thinking about these different mechanisms which you can use to customize LLMs to work with your own data. I hope you found this lesson useful. Thanks for your time.

# 4.6-Personalice los LLM con sus datos

En esta lección, veremos cómo puedes personalizar los modelos LLM para que funcionen con tus propios datos. El marco que vamos a analizar aquí es el siguiente.

Tenemos dos ejes: el horizontal y el vertical. En el eje horizontal se encuentra la optimización del contexto. Aquí proporcionamos información más específica y detallada para que el modelo LLM la utilice. Por ejemplo, información específica del usuario, como los pedidos que ha realizado en un sitio web. En el eje vertical se encuentra la optimización del modelo LLM. Aquí adaptamos el modelo para realizar tareas específicas o trabajar dentro de un dominio determinado. Esto podría ser el ajuste fino del modelo en el ámbito legal.

Por lo tanto, el primer y más obvio punto de partida es la ingeniería rápida. Es la más veloz y eficiente. Permite aprender, probar e iterar rápidamente. Si se necesita más contexto , se puede utilizar algo como la Generación Aumentada de Recuperación. Si se requiere mayor capacidad de seguimiento de instrucciones, se puede recurrir al ajuste fino. Todos estos métodos son aditivos , por lo que es posible que se terminen utilizando todos.

Analicemos cada uno de ellos con un poco más de detalle. Empecemos con la Generación Aumentada de Recuperación. En este caso, el modelo de lenguaje puede consultar bases de conocimiento empresariales, como bases de datos vectoriales, wikis, etc., para proporcionar respuestas fundamentadas. Con "fundamentadas" nos referimos a que el texto generado se basa en un documento si este lo admite.

En el lado izquierdo , se puede ver a una persona conversando con un chatbot. La persona quiere devolver el vestido que acaba de comprar. El chatbot responde: "Sí, claro. Puede devolverlo. Pero necesito consultar la política de devoluciones". A continuación, el chatbot consulta la base de datos de la empresa, donde se indica un plazo de 30 a 90 días y que los artículos no pueden estar en oferta. Luego, conversa con el usuario, explicándole las reglas. El usuario responde : " No sé si mi artículo estaba en oferta".

Entonces, suben un recibo. Se utiliza algún tipo de servicio de visión artificial, que extrae el precio y la fecha de venta del artículo, y luego regresa a una base de datos empresarial, devolviendo los datos e indicando que el artículo se puede devolver. Este es un ejemplo de una implementación RAG. Y se puede ver cómo se fundamentan las respuestas. El chatbot regresa a una base de datos vectorial y determina la política de devoluciones, etcétera, las reglas para dicha política.

RAG no requiere ajustes finos, lo cual representa una gran ventaja. Básicamente, RAG consta de dos componentes principales: la recuperación, que consiste en buscar información relevante en un conjunto de datos para responder a un usuario, como se muestra aquí; y la generación aumentada, que utiliza la información recuperada para ofrecer una respuesta más completa.

Ahora, con RAG, puedes darle a LLM acceso a información privada que de otro modo no tendría, por lo que la política de retorno en este caso cambia. Al permitir que el modelo busque en la base de datos privada y use esa información para generar respuestas, la precisión y la utilidad del modelo cambian drásticamente, como puedes ver en este ejemplo. Este es un ejemplo de implementación de RAG.

Ahora bien, cuando hablamos de ajuste fino e inferencia de LLM, en pocas palabras, el ajuste fino consiste en tomar un modelo fundamental preentrenado, proporcionarle algunos datos personalizados y ajustar ese modelo, entrenarlo, con esos datos personalizados. Y así obtenemos estos datos personalizados.

Para la inferencia, básicamente introducimos un nuevo texto como entrada. El modelo genera una salida basándose en el conocimiento adquirido durante el preentrenamiento y el ajuste fino. Así, responde en función de ambos conocimientos. Este es el proceso de inferencia para el modelo ajustado.

¿Cómo funciona realmente el ajuste fino? Bueno, con el ajuste fino se optimiza el modelo en un conjunto de datos más pequeño y específico del dominio, como se puede ver aquí. Se utiliza el ajuste fino cuando el modelo preentrenado no realiza bien las tareas o cuando se desea enseñarle algo nuevo. Por ejemplo, un dominio único, nuevas habilidades, tareas o un estilo particular.

Así pues, se toma el modelo preentrenado, se le incorpora este conocimiento y se obtiene un modelo personalizado o un modelo ajustado. Un modelo ajustado consiste básicamente en adaptarse a un estilo y tono específicos y aprender palabras y frases propias del dominio, como se muestra aquí.

Ahora bien, no vamos a entrar en conceptos avanzados como la actualización de los pesos, etc. Pero en OCI, existe un método llamado ajuste fino T-Few, que inserta nuevas capas y actualiza selectivamente solo una fracción del peso del modelo, como se puede ver aquí. Al hacer esto, se reduce significativamente el tiempo y el costo del entrenamiento en comparación con la actualización de todas las capas del modelo preentrenado. Este es un concepto más avanzado que trataremos en otros cursos.

La optimización ofrece dos ventajas principales. La primera es que mejora el rendimiento del modelo en tareas específicas. Al personalizarlo con datos específicos del dominio, puede comprender mejor y generar respuestas contextualmente relevantes. Esa es la principal ventaja.

En segundo lugar, también se mejora la eficiencia del modelo. Se reduce el número de tokens y se condensa la experiencia de un modelo grande en uno más pequeño y eficiente. Por lo tanto, existen dos ventajas: una mejora en el rendimiento y otra en la eficiencia del modelo.

La siguiente pregunta, obvia, es si conviene usar uno u otro método y cuándo usar cada uno. Bueno, este es, de nuevo, un marco de trabajo que se puede utilizar. La ingeniería de mensajes es, principalmente, muy sencilla de usar. No requiere capacitación. ¿Y cuándo se debe usar? Cuando el estudiante de derecho ya comprende los temas necesarios para la generación de texto.

Por otro lado, el sistema RAG resulta muy útil cuando los datos cambian rápidamente o cuando se desea mitigar la confusión al contextualizar las respuestas, como se vio en el ejemplo del chatbot. Las ventajas son evidentes: se contextualizan los resultados y se accede a los datos más recientes. La desventaja radica en que requiere una fuente de datos compatible. Si la fuente de datos no es de buena calidad, la recuperación de datos no será óptima. Además, su configuración es más compleja.

Finalmente, se requiere un ajuste fino cuando el modelo LLM no funciona bien en una tarea específica y los datos necesarios son demasiado grandes. En ese caso, se utiliza el ajuste fino. La ventaja, como vimos en la diapositiva anterior, es que mejora el rendimiento y la eficiencia del modelo. Sin embargo, la desventaja es que requiere un conjunto de datos etiquetados, cuya adquisición puede ser costosa y llevar mucho tiempo. Además, requiere más recursos.

Muy bien. Finalmente, permítanme presentarles este marco de referencia, del que hablamos al inicio de esta lección, y analizar cómo podrían utilizarlo. Un recorrido típico se vería así.

Empiezas por la esquina inferior. Tienes una consigna y un marco de evaluación. Luego, determinas cuál es la línea base. A continuación, proporcionas algunos ejemplos breves de pares de entrada/salida que quieres que siga el modelo. Si estás satisfecho con esto, probablemente solo necesites optimizar la consigna y no hacer nada más.

Ahora bien, supongamos que estos breves ejemplos mejoran el rendimiento de su modelo. Ahora puede conectar el modelo a una base de conocimientos empresarial y crear un sistema RAG, como ya comentamos. Así que, en efecto, podría hacerlo. Y sus resultados parecen buenos. Su evaluador es la referencia. Y probablemente se detenga aquí.

Pero si no es así, digamos que la salida del modelo no tiene el formato o el estilo que queremos, entonces puedes ajustar este modelo optimizado con RAG. Podrías optimizarlo y luego ajustarlo. Y tal vez quedes satisfecho.

Pero supongamos que la recuperación con este modelo no es buena. Entonces se puede optimizar la recuperación. Y se puede optimizar el sistema RAG, y así sucesivamente. Se puede ver cómo funciona esto de forma iterativa. Y se terminan utilizando estos tres mecanismos. Por lo tanto, este es un buen marco de referencia para considerar cuando se piensa en la recuperación de ingeniería de mensajes instantáneos, la generación aumentada y el ajuste fino.

Eso es todo. Es un área muy interesante. Este tipo de marco de trabajo resulta muy útil para considerar los diferentes mecanismos que se pueden usar para personalizar los modelos LLM y adaptarlos a los datos propios. Espero que esta lección les haya sido útil. Gracias por su tiempo.

# 4.6 - Personalize os LLMs com seus dados

Nesta lição, vamos analisar como você pode personalizar os LLMs para que funcionem com seus próprios dados. Portanto, a estrutura que discutiremos aqui é a seguinte.

Temos, portanto, dois eixos: o horizontal e o vertical. No eixo horizontal, temos a otimização de contexto. Aqui, fornecemos informações mais específicas e detalhadas para o LLM (Modelo de Aprendizagem Jurídica) utilizar. Um exemplo seria informação específica do usuário, como os pedidos que ele fez em um site. No eixo vertical, temos a otimização do LLM. Aqui, adaptamos o LLM para realizar tarefas específicas ou trabalhar dentro de um domínio determinado. Isso poderia ser o ajuste fino do modelo no domínio jurídico.

Então, o primeiro e mais óbvio ponto de partida é a engenharia de prompts. É a mais rápida, a que oferece maior agilidade. Você pode aprender, testar e iterar rapidamente. Se precisar de mais contexto , pode usar algo como a Geração Aumentada por Recuperação. Se precisar de maior capacidade de seguir instruções, pode usar o ajuste fino. E todos esses métodos são aditivos . Portanto, você pode acabar usando todos eles.

Vamos analisar cada um deles com um pouco mais de detalhes. Começando com a Geração Aumentada por Recuperação, o que fazemos é que o modelo de linguagem pode consultar bases de conhecimento corporativas. Essas bases podem ser bancos de dados vetoriais, wikis, etc., para fornecer respostas fundamentadas. E o que queremos dizer com fundamentadas é que o texto gerado está fundamentado em um documento se o documento suportar o texto.

Então, do lado esquerdo , você pode ver uma pessoa conversando com um chatbot. A pessoa quer devolver o vestido que acabou de comprar. O chatbot responde: "Sim, claro. Você pode devolver. Mas preciso verificar a política de devolução." Em seguida, o chatbot acessa um banco de dados corporativo, consulta a política de devolução, que indica um prazo de 30 a 90 dias e a proibição de itens em promoção. Então, ele conversa com o usuário, informando as regras. O usuário responde : " Não sei se meu item estava em promoção."

Então, eles enviam um recibo. O sistema usa algum tipo de serviço de visão computacional para extrair o preço e a data da venda do item, consultando um banco de dados corporativo, retornando os dados e informando que o item pode ser devolvido. Este é um exemplo de implementação de RAG (Respostas, Ações e Regras). E você pode ver como as respostas são fundamentadas. O chatbot consulta um banco de dados vetorial e verifica a política de devolução, etc., as regras para a política de devolução.

Agora, o RAG não requer ajustes finos. Essa é uma grande vantagem. Basicamente, o que você faz com o RAG é o seguinte: existem dois componentes principais. Um é a recuperação, onde você pesquisa em um conjunto de informações relevantes para responder a um usuário, como você vê aqui. E o segundo é a geração aumentada. Você usa as informações recuperadas para formar uma resposta mais embasada.

Agora, usando RAG, você pode dar ao LLM acesso a conhecimento privado que ele não teria de outra forma, como a política de retorno neste caso. Ao permitir que o modelo pesquise em um banco de dados privado e use essas informações para formular respostas, a precisão e a utilidade do modelo mudam drasticamente, como você pode ver neste exemplo aqui. Este é um exemplo de implementação de RAG.

Agora, quando falamos sobre ajuste fino e inferência em modelos de aprendizado de máquina (LLM), de forma bem simples, o ajuste fino funciona da seguinte maneira: pegamos um modelo básico pré-treinado, fornecemos alguns dados personalizados e ajustamos esse modelo, treinando-o com esses dados personalizados. E assim, obtemos esses dados personalizados.

E então, para a inferência, basicamente fornecemos um novo texto como entrada. O modelo gera uma saída com base no conhecimento que adquiriu durante o pré-treinamento e o ajuste fino. Assim, com base nesse conhecimento, ele responde. Essa é a inferência para o modelo ajustado.

Como funciona o ajuste fino na prática? Bem, o que você faz com o ajuste fino é otimizar o modelo em um conjunto de dados menor e específico do domínio, como você pode ver aqui. Você usaria o ajuste fino quando o modelo pré-treinado não executa bem suas tarefas ou quando você deseja ensiná-lo algo novo. Portanto, pode ser um domínio único. Podem ser novas habilidades, tarefas ou um estilo único.

Então você pega o modelo pré-treinado, incorpora esse conhecimento e, em seguida, obtém um modelo personalizado ou um modelo ajustado. Um modelo ajustado basicamente se adapta a um estilo e tom específicos e aprende palavras e frases específicas do domínio, como mostrado aqui.

Agora, não vamos abordar conceitos avançados como a atualização dos pesos, etc. Mas no OCI, existe um método chamado ajuste fino T-Few, que insere novas camadas e atualiza seletivamente apenas uma fração dos pesos do modelo, como você pode ver aqui. Fazendo isso, o tempo e o custo de treinamento são significativamente reduzidos em comparação com a atualização de todas as camadas de um modelo pré-treinado. Este é um conceito mais avançado que abordaremos em outros cursos.

Existem dois benefícios principais no ajuste fino. O primeiro é a melhoria do desempenho do modelo em tarefas específicas. Ele pode funcionar melhor — porque você está personalizando o modelo para dados específicos do domínio — e gerar respostas contextualmente relevantes com mais precisão. Então, esse é o primeiro benefício.

Em segundo lugar, você também está melhorando a eficiência do modelo. Você está reduzindo o número de tokens e condensando a expertise de um modelo grande em um modelo menor e mais eficiente. Portanto, essas são duas vantagens: melhorar o desempenho e a eficiência do modelo.
